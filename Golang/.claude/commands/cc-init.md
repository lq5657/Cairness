# cc-init

## 用途

`cc-init` 只用于识别项目事实，并回写 `context/project-context.md`。
它不是脚手架安装命令，不是项目体检命令，不是变更提案命令。
它的默认目标是完成“最小可用上下文”，不是一次性补齐完整项目画像。

## 触发场景

适用于：
- 新接入一个已有项目
- `project-context.md` 明显失真或过期
- 需要快速重新确认目录结构、依赖入口、配置入口和测试入口

不适用于：
- 想审查存量项目问题
- 想一次性补齐分层、日志、配置、可观测性、测试策略等完整画像
- 想直接开始实现一个需求
- 想补齐 `.claude/` 脚手架目录

## 输入

命令格式：
- `cc-init`

## 输出

产出：
- 更新 `context/project-context.md`

不产出：
- `changes/<change-id>/`
- `audits/<audit-id>/`
- 示例 change
- 模板目录

展示 checkpoint 表时：
- 必须把状态写入 `结果` 列
- `结果` 仅允许填写 `✅`、`❌`、`⚠️` 或 `N/A`
- 不要在 `检查项` 列使用 `[ ]` / `[x]` 代替结果

## 必守边界

- 只做项目事实识别，不做问题审查
- 只回写 `project-context.md`，不创建脚手架资产
- 不得因为缺少脚手架而创建 `rules/`、`knowledge/`、`changes/`、`audits/`
- 不得因为缺少 examples/templates 而补目录
- 不得把 `cc-init` 扩展为 `cc-inspect-codebase`
- 不得基于猜测伪造项目事实

## 最小识别原则

`cc-init` 默认只完成“最小可用”的项目事实识别，不追求一次性填完整个 `project-context.md`。

最小可用事实包括：
- 项目模式（已有项目 / 新项目）
- 应用名 / 简介
- `go.mod` 或等价依赖入口
- 根目录关键结构
- 模块入口或启动入口
- 配置入口
- 日志入口（若可低成本确认）
- 测试入口（若可低成本确认）
- 已确认事实与待确认事项

以下内容在首次 `cc-init` 中默认允许留为“待确认”：
- 实际分层与调用关系
- 团队规范
- 可观测性细节
- 环境差异
- 领域高风险点
- 测试分层策略细节

若证据不足，必须标记“待确认”，不得为了补齐这些字段继续扩大读取范围。

## 允许读取的范围

允许读取：
- 仓库根目录最小结构
- `go.mod` 或等价依赖入口
- 一个主入口文件或启动入口
- 一个配置入口文件
- 一个日志相关入口文件（若存在）
- 最多两个测试样本文件
- 现有 `context/project-context.md`

禁止读取：
- 大量业务代码正文
- 全仓逐文件扫描
- 审查导向的深度代码阅读
- 为补齐高解释成本字段而持续扩大读取范围

## 默认执行流程

1. 确认 `.claude/` 脚手架是否存在
2. 若脚手架缺失，停止并提示维护者先安装 harness
3. 识别最小项目事实：
   - 根目录关键结构
   - 模块入口
   - 依赖管理入口
   - 配置入口
   - 日志入口（若可低成本确认）
   - 测试入口（若可低成本确认）
4. 将无法低成本确认的内容标记为“待确认”
5. 更新 `context/project-context.md`
6. 结束，不自动进入 `cc-enrich-context`、`cc-propose` 或 `cc-inspect-codebase`

## 失败处理

若以下情况出现，必须停止并说明：
- `.claude/` 脚手架缺失
- 项目结构不足以支撑事实识别
- 关键事实无法确认
- 当前仓库状态混乱，无法可靠判断事实

## 执行后建议

执行完成后，下一步通常是二选一：
- 若需要补齐更完整的项目画像：执行 `cc-enrich-context`
- 若需要做存量项目体检：执行 `cc-inspect-codebase <mode>`
- 若已有明确需求：执行 `cc-propose <需求描述>`

## 需要加载的附加文件

- `checkpoints/cc-init.md`
- `context/project-context.md`
