### Release And Rollback

#### 约束等级

- 🚫 强制 — 违反则停止执行
- ⚠️ 警告 — 触发时需人工确认或补充说明
- ✅ 验证 — 完成后必须检查

#### 使用范围

当变更涉及以下任一场景时，必须应用本规则：
- 数据库 schema、回填、配置、接口契约等高风险变更
- 需要灰度、开关、分批放量的能力
- 涉及回滚成本高或回滚路径不对称的实现
- 发布后必须重点观察成功率、延迟、错误率或业务指标的变更

#### 1. 基本原则

- 变更的交付边界不止是“代码合并”，还包括“上线策略可执行、回滚路径可说明、观察窗口可验证”。
- `cc-propose` 时必须明确本次变更的发布方式和回滚方式。
- 回滚不能默认等于“git revert”；数据库、配置、接口和开关往往需要额外恢复动作。
- `cc-review` 必须检查：spec 中声明的发布与回滚方案是否与实际实现、依赖和风险相匹配。

#### 2. 🚫 强制规则

- 禁止高风险变更完全没有回滚路径说明。
- 禁止依赖 feature flag 或灰度开关却不说明启用/关闭策略。
- 禁止把数据库不可逆变更、breaking contract 变更当作“随时可直接回滚”的普通发布。
- 禁止发布观察窗口完全缺失，尤其是涉及资金、状态、配置、消息、异步任务的变更。

#### 3. ⚠️ 需要显式说明的场景

- 灰度发布、分批放量、地域切换、租户分批
- feature flag / kill switch / 降级开关
- 数据库 expand / migrate / contract
- 接口 breaking change 或客户端迁移窗口
- 回滚需要数据修复、开关回拨、配置恢复、脚本补偿

上述场景至少要说明：
- 发布方式
- 启用/关闭条件
- 回滚路径
- 发布后观察窗口
- 失败触发条件

#### 4. 推荐做法

- 高风险变更优先采用“先兼容、再切流、后清理”的渐进式上线方式。
- 回滚路径要区分：代码回滚、配置回滚、数据回滚、开关回拨。
- 若只能前滚（roll-forward），必须明确说明原因和止损手段。
- 发布后观察窗口至少要绑定一组可量化信号：成功率、延迟、错误率或关键业务指标。

#### 5. ✅ 最低验证要求

- [ ] `spec.md` 已记录发布方式、开关策略、回滚路径、观察窗口
- [ ] `log.md` 可记录发布决策与回滚动作
- [ ] 若涉及高风险变更，已说明为什么当前发布方式足够安全
- [ ] 若无法直接回滚，已说明前滚/补偿方案
