### Database Changes

#### 约束等级

- 🚫 强制 — 违反则停止执行
- ⚠️ 警告 — 触发时需人工确认或补充说明
- ✅ 验证 — 完成后必须检查

#### 使用范围

当变更涉及以下任一场景时，必须应用本规则：
- 新增、删除、重命名、修改表/列/索引
- 新增 migration 脚本或手工 DDL
- 数据回填、批量修复、历史数据重算
- 读写兼容窗口、双写、灰度迁移、schema 收缩

#### 1. 基本原则

- schema 变更必须通过可审查的 migration 或等价变更脚本交付，不得仅口头说明“上线时手改库”。
- 任何破坏性数据库变更都必须显式说明兼容窗口与回滚路径。
- 数据回填与业务逻辑改动默认视为高风险；若能拆分，优先拆成独立 task 或独立 change。
- `cc-review` 必须检查：spec 中声明的数据变更、实际 migration、代码兼容策略三者是否一致。

#### 2. 变更分类

| 类型 | 示例 | 默认风险 |
|------|------|----------|
| Expand | 新增可空列、新增兼容索引 | 中 |
| Migrate | 回填历史数据、双写、数据搬迁 | 高 |
| Contract | 删除列、收紧约束、切换读路径、删除旧索引 | 高 |

#### 3. 🚫 强制规则

- 禁止在未说明 migration 路径时修改依赖新 schema 的代码。
- 禁止将“新增非空列且无默认值”“直接删列”“直接改列类型”视为低风险改动。
- 禁止未说明幂等性、批量范围、失败恢复方式就执行数据回填。
- 禁止在未确认兼容窗口结束前删除旧字段读写逻辑。
- 禁止将数据库破坏性变更与无关业务改动混在同一个 task 中提交。

#### 4. ⚠️ 需要显式说明的场景

- 新增非空约束、唯一约束、外键、默认值变更
- 大表加索引、索引重建、字段类型调整
- 历史数据回填、批量修复、离线脚本
- 读写双写、双读、切流、灰度迁移
- 会影响旧版本服务实例、旧消息消费者或回滚版本的 schema 变更

上述场景至少要说明：
- 兼容窗口
- 回滚路径
- 失败后恢复方式
- 是否需要人工审查

#### 5. expand / migrate / contract 建议

**Expand：**
- 优先新增兼容字段或索引，不立即删除旧结构
- 新代码上线前，旧代码应仍可正常读写

**Migrate：**
- 回填脚本要说明执行范围、批次、幂等性、停止条件
- 需要写明“失败后重跑是否安全”

**Contract：**
- 只有在旧代码路径、旧消费者、回滚版本都不再依赖旧 schema 时，才允许收缩
- 收缩前必须说明兼容窗口已结束

#### 6. ✅ 最低验证要求

- [ ] `spec.md` 已记录 migration / 兼容窗口 / 回滚路径
- [ ] `tasks.md` 已将 schema 变更、回填、业务代码切换拆出清晰顺序
- [ ] 实际 migration 或脚本路径可定位
- [ ] 若涉及回填，已说明幂等性、批次策略和失败恢复方式
- [ ] 若涉及破坏性变更，已说明旧版本兼容性与回滚影响
