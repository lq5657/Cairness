### Harness 试点验收清单

用于维护者判断：当前这套 Claude Code Harness 是否已经可以在真实团队或真实项目中试点。

#### 1. 发布前最低门槛

以下项目全部满足，才建议进入团队试点：

| 项目 | 验收标准 | 状态 |
|------|----------|------|
| 生命周期清晰 | `propose -> apply -> review -> done` 已定义清楚，且模板字段一致 | [ ] |
| 文档契约稳定 | `spec.md`、`tasks.md`、`log.md`、`test-spec.md`、`review.md` 结构已固定 | [ ] |
| reviewer 可落地 | 两个 reviewer 的输出可直接映射到 `review.md`，不依赖自由发挥 | [ ] |
| 高风险规则可判定 | 资金、状态、权限、数据库、接口、配置、发布回滚规则已有明确触发条件和最低证据 | [ ] |
| 至少一个端到端样例 | `changes/examples/` 下存在完整示例，且与当前规则一致 | [ ] |
| README 可引导新人 | 新维护者只读 README 即可找到模板、样例、核心流程 | [ ] |

#### 2. 试点前环境检查

建议在正式试点前，用一个低风险真实项目完成以下检查：

| 项目 | 通过标准 | 状态 |
|------|----------|------|
| Claude Code 能读取规则 | 会话开始时能稳定加载 `.claude/CLAUDE.md` 和 `rules/` | [ ] |
| `/init` 可填充上下文 | 能输出真实目录结构、依赖和团队约定，而不是套模板 | [ ] |
| `/propose` 能生成可用文档 | 生成的 spec/tasks 不需要大面积人工重写 | [ ] |
| `/apply` 不会频繁自锁 | 开发过程中不会因规则冲突卡死，需要的偏差回写路径可执行 | [ ] |
| 并发治理可操作 | 维护者能稳定判断文件级/调用链冲突，并知道记录到哪里 | [ ] |
| 高风险治理可落地 | 数据库、接口、配置、发布回滚字段在真实 change 中都能填写清楚 | [ ] |
| `/review` 结果稳定 | 两个 reviewer 对同类问题给出的结论基本一致 | [ ] |
| auto-commit 可接受 | 团队能接受一个 task 一个 commit 的粒度和 message 格式 | [ ] |

#### 3. 首轮试点建议范围

第一轮不要直接上高风险需求，建议限定范围：

| 维度 | 建议 |
|------|------|
| 业务复杂度 | 先选 CRUD、查询、非资金类写接口 |
| 参与人数 | 1-2 名维护者 + 1 个真实项目 |
| 变更规模 | 1-3 个 task，单次改动不超过 15 个文件 |
| 风险控制 | 不接支付、账务、权限改造、核心状态机重构 |
| 验证方式 | 保留人工 review，不把 harness 直接当自动化审批器 |

#### 4. 试点中必须观察的信号

如果出现以下现象，说明规范还不够稳定：

| 信号 | 说明 | 建议动作 |
|------|------|----------|
| AI 经常卡在流程冲突 | 规则之间仍有互相阻断 | 先修规则，不要继续加模板 |
| spec/tasks 经常被人工重写 | `/propose` 产物质量不够 | 回头收紧模板和提问策略 |
| review 结论不一致 | reviewer 输出契约仍然不稳 | 补更多示例和判定样例 |
| 变更日志没人维护 | 文档负担过重或价值不明显 | 缩减字段或明确最小填写要求 |
| auto-commit 被频繁绕过 | commit 粒度不适配团队节奏 | 调整 task 粒度或 commit 约束 |
| 高风险字段长期留空 | 新规则无法在真实项目落地 | 先补样例，再收紧必填范围 |

#### 5. 试点通过标准

满足以下条件，才建议扩大使用范围：

| 条件 | 说明 | 状态 |
|------|------|------|
| 连续 3 个真实变更跑通 | 从 `/propose` 到 `/review` 都能完成 | [ ] |
| 无严重流程冲突 | 没有出现阻塞研发的硬性自锁 | [ ] |
| reviewer 结论可复核 | 人工维护者认为审查结果大体可信 | [ ] |
| 文档维护成本可接受 | 团队认为额外文档工作量可承受 | [ ] |
| 规则无需频繁热修 | 一个试点周期内不需要每天改规则 | [ ] |
| 高风险变更有样例可参考 | 至少已有数据库/接口/配置/发布回滚中的一类示例 | [ ] |

#### 6. 不建议立即推广的情况

- 还没有真实项目跑通过一次完整流程
- 维护者自己都无法快速解释每个命令的输入输出
- `project-context.md` 在不同项目上仍然大面积失真
- 团队无法接受 auto-commit 或功能分支要求
- 当前项目本身缺少基本构建和测试能力，导致 harness 无法提供验证证据

#### 7. 维护者行动建议

如果准备开始试点，推荐顺序：

1. 用 `changes/examples/user-create-api/` 给团队过一遍完整流程。
2. 选一个低风险真实需求，先人工陪跑一次。
3. 记录试点中最常见的 3 个卡点，优先修规则，不优先加新功能。
4. 至少跑满 3 个真实变更，再决定是否推广到更高风险模块。
