### `cc-propose` 回归评测样例

用于维护者在修改 `cc-propose`、`checkpoints/cc-propose.md` 或相关提问策略后，快速验证命令是否仍能正确处理绿地需求、存量需求和需求澄清边界。

#### 使用方式

1. 选一个样例输入给运行中的 harness。
2. 只观察 `cc-propose` 的前半段行为，不要求真的生成最终 `spec.md`。
3. 按本文“必过信号 / 失败信号 / 评分点”打分。
4. 若未通过，先修 `commands/cc-propose.md` 或 `checkpoints/cc-propose.md`，再回归。

---

#### 通用评分维度

每个样例至少检查以下 8 项：

| 维度 | 通过标准 |
|------|----------|
| 命令识别 | 能进入 `cc-propose` 主流程，而不是误判为 `cc-init` / `cc-preflight` / 其他命令 |
| 项目形态判断 | 能识别当前是绿地/空项目还是存量项目增强 |
| 需求发现优先级 | 绿地场景优先做 Discovery，不先切成技术问卷 |
| 问题贴合度 | 第一轮问题直接围绕用户原始需求中的核心名词、动作和场景 |
| 理解回显 | 每轮追问前或追问后，能输出当前理解摘要 |
| 技术问题后置 | 在需求未收敛前，不优先追问技术栈、运行形态、存储、模型接入 |
| 范围意识 | 能识别本次要做 / 本次不做，不无限扩写需求 |
| 收敛能力 | 能在合理轮数内把问题收敛到可提案状态，或明确进入 `brainstorm-needed` |

建议打分：
- `2`：完全符合
- `1`：部分符合但有明显跑偏
- `0`：未体现或明显错误

总分建议：
- `14-16`：通过
- `10-13`：可接受，但需要观察
- `<10`：不通过，必须回修

## Case 1：绿地角色扮演练习平台

**目标**：验证空项目绿地需求不会被误导到 `cc-init`、技术问卷或工程接入建议。

**用户输入**：

```text
cc-propose 我要实现一个能够和用户进行对话的智能体，它能作为不同的角色，模拟指定场景下的对话
```

**场景前提**：
- 仓库为空或几乎为空
- 没有 Go 源文件
- `project-context.md` 仍是模板或未初始化

**必过信号**：
- 不把主响应退化成“空项目，建议先执行 cc-init”
- 第一轮问题围绕“不同角色”“指定场景”“主要用途”展开
- 能追问用户真实目标，如训练、教育、内容创作、业务模拟
- 每轮回答后能总结：目标、目标用户、典型场景、成功标准
- 技术问题后置到需求较清晰之后

**失败信号**：
- 一上来把重点放在技术栈、模型选择、运行形态、存储方式
- 把“没有代码可研究”直接解释成“无法继续 propose”
- 只输出 `light-clarify` / `brainstorm-needed` 标签，没有用户可感知的理解摘要
- 直接推荐 `cc-init` 作为继续提案前的前置步骤

**评分补充**：
- 若第一轮提问中 50% 以上是技术实现问题，本例直接判定不通过

## Case 2：绿地但需求已相对明确

**目标**：验证绿地场景下不会为了“流程完整”强行拉长 Discovery。

**用户输入**：

```text
cc-propose 我需要做一个内部 Web 管理台，管理员可以查看订单列表、筛选状态、导出 CSV，不需要编辑订单
```

**场景前提**：
- 新项目
- 无历史代码

**必过信号**：
- 先确认目标用户、使用场景、成功标准和边界
- 能快速识别“管理员、订单列表、状态筛选、CSV 导出、不支持编辑”这些核心点
- 若缺口不大，进入 `light-clarify` 而不是硬拖成长对话
- “本次不做”至少包含“编辑订单”或其他用户明确排除的能力

**失败信号**：
- 明明需求已很具体，仍然长时间停留在泛泛 brainstorm
- 忽略用户明确说过的排除项
- 先问 React / Vue / Gin / Echo / 数据库等技术问题

## Case 3：存量项目新增能力

**目标**：验证存量项目仍然保留“先 Research”的强项，不会被新 Discovery 规则削弱。

**用户输入**：

```text
cc-propose 给现有用户服务增加批量冻结用户接口，要求保留审计日志和原有权限校验
```

**场景前提**：
- 仓库存在用户服务代码
- `project-context.md` 已有基础事实

**必过信号**：
- 先做本地 Research，识别现有冻结/权限/审计链路
- 提问聚焦业务规则缺口，例如冻结条件、批量失败语义、审计字段
- 不把它错误切回绿地 Discovery 模式长聊产品愿景
- 能把“保留原有权限校验”识别为既有模式沿用要求

**失败信号**：
- 不查代码就直接提方案
- 完全忽略“现有用户服务”“原有权限校验”这些存量上下文信号
- 把问题聊成产品定位讨论

## Case 4：绿地需求但用户语言很抽象

**目标**：验证 `brainstorm-needed` 分支是否健康，不会直接坠入技术方案。

**用户输入**：

```text
cc-propose 我想做一个更懂用户的沟通系统，体验要比现在的方案好很多
```

**场景前提**：
- 新项目
- 用户描述明显抽象

**必过信号**：
- 明确进入 `brainstorm-needed` 或等价短收敛
- 追问“更懂用户”“体验更好”分别具体指什么
- 能把抽象词拆成可回答的问题，而不是泛泛追问“技术栈是什么”
- 收敛后能回到提案主流程，而不是停留在无边界闲聊

**失败信号**：
- 接受抽象描述，不继续具体化
- 直接跳到系统架构、模型、数据库
- 脑暴过程产出独立长期文档，脱离 `cc-propose`

---

## Case 5：空项目但用户明确要求先做工程脚手架

**目标**：验证新规则不会让所有空项目都被当成产品 Discovery，仍能识别用户确实在问工程初始化。

**用户输入**：

```text
cc-propose 我想先搭一个 Golang 后端脚手架，包含 Gin、GORM、JWT 登录、基础配置和日志
```

**场景前提**：
- 新项目
- 用户关注点明显偏工程脚手架

**必过信号**：
- 仍然先确认边界和目标，但可以更快进入方案收敛
- 问题聚焦脚手架用途、预期模块、是否作为团队模板、哪些能力本次不做
- 不会机械地套用长 Discovery 访谈

**失败信号**：
- 明明是工程脚手架需求，却还长时间追问产品愿景、用户画像
- 或者相反，一上来只报技术栈清单，不做任何边界确认

---

## 维护建议

- 每次修改 `commands/cc-propose.md` 或 `checkpoints/cc-propose.md` 后，至少回归 Case 1、Case 3
- 若修改了 `brainstorm-needed` 判定，再额外回归 Case 4
- 若修改了空项目策略，再额外回归 Case 1、Case 5
- 若回归失败，优先修命令规则和 checkpoint，不优先补 README 解释
