### 编码规范

#### 1. 命名 (结合 Go 最佳实践)

* 文件命名：全小写，下划线分隔（例如 `user_service.go`）。
* 结构体/接口名：大驼峰，见名知意；接口通常以 `er` 结尾（如 `Reader`, `Writer`） 。
* 变量/函数名：驼峰命名，首字母大小写严格控制包内外的可见性（导出用大驼峰，非导出用小驼峰） 。
* 常量：大驼峰或全大写下划线分隔 。
* 测试类：与被测文件同名加 `_test.go` 后缀，测试函数以 `Test` 开头 。
* 禁止拼音、中英混拼命名 。

#### 2. 错误与异常处理 (结合 Go 最佳实践)

* 业务错误：必须使用自定义 Error 类型（如透传错误码），通过返回值传递，禁止滥用 `panic` 抛出异常 。
* 错误透传：使用 `fmt.Errorf("... %w", err)` 包装错误以保留底层错误堆栈。
* 系统崩溃兜底：关键的独立 Goroutine 入口必须使用 `defer recover()` 进行兜底。
* 禁止吞掉错误（禁止写 `_ = err` 或只打日志不 return） 。

#### 3. 日志

* Handler/Controller 入口打 INFO，含请求关键参数 。
* 错误打 ERROR，含完整日志上下文和 traceID 。
* 禁止在日志中打印用户敏感信息 。

#### 4. 其他

* 写接口必须考虑幂等 。
* 涉及并发场景必须说明同步策略（如 `sync.Mutex` 或 `channel`） 。
* 魔法值必须定义为常量 。
