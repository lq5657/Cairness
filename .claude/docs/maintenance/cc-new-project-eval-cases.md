### `cc-new-project` 回归评测样例

用于维护者在修改 `cc-new-project`、相关模板或项目级路由规则后，快速验证命令是否仍能正确完成“新项目定义 -> MVP 路线图 -> 首批 change backlog”的闭环。

#### 使用方式

1. 选一个样例输入给运行中的 harness。
2. 观察 `cc-new-project` 是否保持项目级视角，而不是滑回 `cc-propose` 或直接进入编码。
3. 按本文“必过信号 / 失败信号 / 评分点”打分。
4. 若未通过，优先修 `commands/cc-new-project.md`、`checkpoints/cc-new-project.md`、项目级模板和路由规则。

---

#### 通用评分维度

| 维度 | 通过标准 |
|------|----------|
| 命令识别 | 能进入 `cc-new-project` 主流程，而不是误判为 `cc-propose` |
| 项目定义 | 能明确目标、用户、场景、成功标准 |
| MVP 范围 | 能明确“本期必须做 / 后放 / 不做” |
| 灰区收敛 | 能推动用户明确关键实现偏好和灰区决策 |
| 技术方向 | 能在合适时机讨论运行形态、模块和技术方向 |
| 路线图 | 能形成阶段性 roadmap，而不是只给一堆功能点 |
| Backlog | 能给出首批推荐 change backlog |
| 生命周期边界 | 不会直接跳进 `cc-apply` 或把项目级定义写进 `.cc/changes/` |

---

## Case 1：模糊新产品想法

**用户输入**：

```text
cc-new-project 我想做一个和用户对话的角色扮演练习平台
```

**必过信号**：
- 先收敛目标、用户和场景
- 明确 MVP，例如角色配置、对话练习、结果反馈
- 形成首批 change backlog，而不是直接写代码计划

**失败信号**：
- 退化为“建议先执行 cc-init”
- 不收敛 MVP，直接进入技术实现清单

---

## Case 2：项目目标清晰但技术方向未定

**用户输入**：

```text
cc-new-project 我要做一个内部管理台，先支持订单列表、状态筛选和 CSV 导出
```

**必过信号**：
- 先收敛产品范围，再讨论运行形态与技术方向
- 能形成 Phase 0 / Phase 1 路线图
- 能推荐首批 change

**失败信号**：
- 一开始就只问框架和数据库
- 没有形成 roadmap

---

## Case 3：把 `cc-new-project` 错用成 change 提案

**用户输入**：

```text
cc-new-project 给现有用户服务增加冻结接口
```

**必过信号**：
- 能明确指出这更像已有项目中的 change
- 建议改用 `cc-propose`

**失败信号**：
- 明明是存量项目变更，仍强行走项目定义流程

---

## 维护建议

- 每次修改 `cc-new-project` 或项目级模板后，至少回归 Case 1、Case 3
- 若修改了技术方向讨论策略，再额外回归 Case 2
