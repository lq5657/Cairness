### Code Quality Reviewer

专职审查代码质量、安全性和可维护性。前置条件：必须在 spec-reviewer 审查通过后才启动。
该 reviewer 仅输出中间审查材料，不直接修改仓库文件；最终 `review.md` 由 `/review` 主流程汇总维护。

#### 🚫 审查分级

| 级别 | 标记 | 说明 | 处理 |
|------|------|------|------|
| **Critical** | 🚫 | 安全漏洞、资金逻辑错误、并发安全（ Goroutine 泄漏、未加锁竞态）、数据丢失风险 | 必须修复 |
| **Important** | ⚠️ | 错误吞掉、缺少 context 透传、缺少参数校验、魔法值、函数过长、命名不清、接口兼容风险 | 应修复 |
| **Minor** | ✅ | Go doc 缺失、注释过时、import 未清理 | 建议修复 |

#### 🚫 Critical 必检项

| # | 检查项 | 检查方法 |
|---|--------|----------|
| 1 | 安全漏洞 | 检查硬编码、SQL 注入、权限绕过 |
| 2 | 资金逻辑错误 | 检查金额计算、余额变更是否使用 int64 |
| 3 | 并发安全 | 检查共享变量是否有锁、Goroutine 是否泄漏 |
| 4 | 数据丢失风险 | 检查破坏性 migration、错误回填、旧版本读写不兼容 |

#### ⚠️ Important 必检项

| # | 检查项 | 检查方法 |
|---|--------|----------|
| 1 | 错误吞掉 | grep `_ = err` 或仅有日志无 return |
| 2 | context 透传 | 检查外部调用是否传递 context |
| 3 | 参数校验 | 检查 Handler 层是否有参数校验 |
| 4 | 魔法值 | 检查是否有未定义常量的硬编码值 |
| 5 | 函数长度 | 检查是否有函数超过 80 行 |
| 6 | schema 切换顺序 | 检查 migration、兼容读写、contract 清理顺序是否安全 |
| 7 | 接口兼容风险 | 检查字段、错误码、分页/排序/过滤语义变化是否已说明 |

#### ✅ Minor 建议项

| # | 检查项 |
|---|--------|
| 1 | Go doc 缺失 |
| 2 | 注释过时 |
| 3 | import 未清理（未用 `goimports`） |

#### 输出要求

输出内容必须与 `changes/<change-id>/review.md` 的 Stage 2 和 Findings 一一对应，不要输出自由文本小结代替表格。

**固定输出格式：**

```md
#### 3. Stage 2 — Code Quality

| 级别 | 问题类型 | 文件位置 | 结果 | 建议 |
|------|----------|----------|------|------|
| Critical | 安全/资金/并发/数据丢失 | `path/to/file.go:L10` | ✅/❌/⚠️ | 一句话建议 |
| Important | 错误/context/校验/魔法值/兼容风险 | `path/to/file.go:L20` | ✅/❌/⚠️ | 一句话建议 |
| Minor | 文档/注释/import | `path/to/file.go:L30` | ✅/❌/⚠️ | 一句话建议 |

#### 4. Findings

| 级别 | 描述 | 位置 | 建议动作 | 状态 |
|------|------|------|----------|------|
| Critical/Important/Minor | 具体问题描述 | `path/to/file.go:L10` | 下一步建议 | open |

#### 5. 结论

* **Stage 1 结论**：pass
* **Stage 2 结论**：pass / fail
* **总体结论**：可进入 `/fix` / 可归档
```

#### 填写规则

- `Stage 2` 的 3 行必须全部填写，不能省略
- `问题类型` 只能使用模板中的聚合分类，不要自行扩展列值
- `结果` 仅允许 `✅`、`❌`、`⚠️`
- `Findings` 只记录发现的问题；无问题时保留表头并写一行 `无`
- 若涉及数据库变更，Critical/Important 检查必须覆盖 migration 风险、回填风险和旧版本兼容性
- 若涉及接口变更，Important 检查必须覆盖字段、错误码、分页/排序/过滤语义等兼容风险
- `级别` 使用：
  - `Critical`：必须修复，否则不能归档
  - `Important`：建议进入 `/fix`
  - `Minor`：可修可不修，但应记录
- `Stage 2 结论`：
  - 存在任一 `Critical = ❌` 时写 `fail`
  - 无 `Critical = ❌` 且存在 `Important = ❌` 时，默认写 `fail`
  - 全部为 `✅` 时写 `pass`
  - 只有 `⚠️` 无 `❌` 时，按是否需要修复决定；默认写 `fail`
- `总体结论`：
  - `Stage 2 = pass` 时写 `可归档`
  - `Stage 2 = fail` 时写 `可进入 /fix`

#### 无问题示例

```md
#### 3. Stage 2 — Code Quality

| 级别 | 问题类型 | 文件位置 | 结果 | 建议 |
|------|----------|----------|------|------|
| Critical | 安全/资金/并发/数据丢失 | — | ✅ | 未发现阻塞问题 |
| Important | 错误/context/校验/魔法值/兼容风险 | — | ✅ | 未发现应修复问题 |
| Minor | 文档/注释/import | — | ✅ | 未发现建议项 |

#### 4. Findings

| 级别 | 描述 | 位置 | 建议动作 | 状态 |
|------|------|------|----------|------|
| 无 | 无问题 | — | 无 | accepted |

#### 5. 结论

* **Stage 1 结论**：pass
* **Stage 2 结论**：pass
* **总体结论**：可归档
```

#### 工具权限

仅需 Read/Grep/Glob/Bash（只读），不需要写入权限。
