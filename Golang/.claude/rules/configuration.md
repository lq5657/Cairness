### Configuration Management

#### 约束等级

- 🚫 强制 — 违反则停止执行
- ⚠️ 警告 — 触发时需人工确认或补充说明
- ✅ 验证 — 完成后必须检查

#### 使用范围

当变更涉及以下任一场景时，必须应用本规则：
- 新增、删除、重命名配置项或环境变量
- 修改配置默认值、必填性、解析方式或作用域
- 新增 feature flag、灰度开关、降级开关
- 修改密钥、凭证、连接串、第三方接入配置来源
- 依赖不同环境（dev、test、staging、prod）出现差异化行为

#### 1. 基本原则

- 配置项必须被显式声明，不能只存在于代码读取逻辑里。
- 默认值必须说明是否安全；“方便本地跑起来”不等于“适合生产默认值”。
- 配置读取应集中在启动期或依赖注入层，不应在业务深处散落读取环境变量。
- `cc-review` 必须检查：spec 中声明的配置契约、实际代码读取点和默认行为是否一致。

#### 2. 🚫 强制规则

- 禁止新增配置项但不在 `spec.md` 中记录名称、默认值、必填性和影响范围。
- 禁止在业务逻辑深处直接散读 `os.Getenv` 或等价环境读取。
- 禁止将生产敏感值写入源码、模板、测试夹具或示例配置。
- 禁止修改关键配置默认值而不说明对现网或各环境的影响。
- 禁止删除仍被旧部署流程、旧任务或回滚版本依赖的配置项。

#### 3. ⚠️ 需要显式说明的场景

- 配置默认值变化
- 配置从可选变必填，或从必填变可选
- 配置作用域变化：全局 → 局部，或局部 → 全局
- 不同环境行为分叉
- feature flag、新开关、灰度参数
- 密钥来源、轮转方式、注入方式变化

上述场景至少要说明：
- 配置名
- 默认值与是否安全
- 必填性
- 生效范围
- 环境差异
- 回滚影响

#### 4. 推荐做法

- 在配置结构体、加载函数或启动模块集中完成解析和校验。
- 新增配置项时，优先提供显式默认值或启动期失败策略，不要隐式兜底。
- feature flag 要说明启用条件、关闭条件和清理计划。
- 若依赖平台注入（Kubernetes Secret、CI/CD 变量、Vault 等），应在 spec 或 project-context 中记录来源。

#### 5. ✅ 最低验证要求

- [ ] `spec.md` 已记录新增或变更配置项及其默认值、必填性、环境差异
- [ ] `context/project-context.md` 的“配置管理”约定已反映当前项目真实做法
- [ ] 配置读取点、注入点、校验点可定位
- [ ] 若涉及敏感配置，来源与脱敏策略已说明
- [ ] 若涉及开关，已说明启用/关闭策略与回滚影响
