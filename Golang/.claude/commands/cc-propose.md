# cc-propose

## 用途

创建变更提案，产出 `changes/<change-id>/spec.md` 与 `tasks.md`。

## 命令格式

- `cc-propose <需求描述>`

展示 checkpoint 表时：
- 必须把状态写入 `结果` 列
- `结果` 仅允许填写 `✅`、`❌`、`⚠️` 或 `N/A`
- 不要在 `检查项` 列使用 `[ ]` / `[x]` 代替结果

## 执行流程

1. Research：优先读代码、查链路、识别现有实现与项目约定
2. 上下文充分性判断：判断本地代码、上下文和既有约定是否足以支撑方案收敛
3. 外部 Research（条件触发）：若是绿地项目、核心模块从零设计，或问题域存在成熟行业实践但本地缺少可参考实现，则补充受控外部调研，提炼候选方案与权衡
4. 问题重述：先向用户重述目标、边界、已知约束与当前理解
5. 提问澄清：逐个提出高价值问题并等待回答，只问会影响 spec 正确性、范围边界、任务拆分的事项
6. 方案比较：至少给出推荐方案和一个放弃方案，并说明采用/放弃原因
7. 范围冻结：完成 YAGNI 裁剪，明确“本次要做 / 本次不做”
8. 验证映射：为主要功能点/风险点声明最低验证等级、证据类型与对应 task 承接方式
9. 生成 Spec：重点补齐背景、功能点、风险、方案比较、待澄清与验证映射
10. 生成 Tasks：按最小可执行单元拆 task，而不是仅按文件罗列，并确保 task 验证步骤可回溯到验证映射
11. HARD-GATE：等待用户确认再进入 `cc-apply`

## 强制边界

- `spec.md` 的“待澄清”章节全部解决前，禁止进入 `cc-apply`
- 创建提案时必须检查是否与现有 change 存在文件级或链路级冲突
- 若存在明显依赖，必须在 `spec.md` 中记录 `depends_on`
- 若涉及 DB、API、配置、可观测性、测试、发布等专题，必须增量读取对应规则
- 未做最小必要 Research 前，不得直接提出泛化澄清问题
- 当本地上下文不足以支撑方案收敛，且问题域存在成熟行业实践时，必须先做受控外部 Research，再生成最终方案比较
- 外部 Research 只用于提炼候选方案、关键权衡和适配条件；正式结论必须写入 `spec.md` 的“方案比较”“技术决策”和“本次不做”，不得以外部资料替代 change 提案
- 未完成主要需求项/风险点到验证等级与证据类型的映射前，不得生成最终版 `tasks.md`
- `spec.md` 中的最低验证等级、验证证据要求与 `tasks.md` 的验证步骤必须可追溯，不得互相脱节
- 未完成范围冻结前，不得生成最终版 `tasks.md`
- 若仍存在影响 task 拆分的关键未决问题，只能产出草案，保持 `status: propose`，不得宣称提案已就绪

## 失败处理

- 若提问后仍有未解决澄清项，保持 `status: propose`
- 若已产出部分 spec/tasks 但信息不足以继续，记录为 `partial` 或 `blocked`
