### 编码规范

#### Skill Anatomy

**When To Use**
- 任何 `cc-apply` / `cc-fix` 涉及 Go 源码、测试、错误处理、日志、并发或依赖变更。
- `cc-review` 检查实现质量、命名、错误路径、日志和可维护性风险。
- 新增接口、goroutine、第三方依赖或业务写操作。

**When Not To Use**
- 不用它替代需求、验证等级、数据库、API 或发布规则。
- 纯文档、纯 Harness 元数据或生成代码变更可只检查是否误改手写逻辑。
- 不用局部编码偏好覆盖项目已确认的工程约定。

**Process**
1. 定位本次变更触及的手写代码、测试、日志和依赖。
2. 按命名、错误处理、日志、并发、抽象和依赖规则逐项检查。
3. 对业务错误、敏感日志、`panic`、吞错、无退出机制 goroutine 等问题先修正再继续。
4. 若必须例外，记录原因、影响范围和验证证据。

**Common Rationalizations**

| Rationalization | Why It Is Invalid | Required Response |
|-----------------|-------------------|-------------------|
| "现有代码也这么写。" | 既有不一致不是新增问题的授权。 | 遵守当前规则，或记录兼容旧代码的最小例外。 |
| "只是临时日志/临时 helper。" | 临时代码经常进入主干并形成生产风险。 | 按正式代码处理，或明确删除路径和验证结果。 |
| "这个错误不会发生。" | 未处理错误会隐藏真实失败点。 | 返回、包装或显式记录不可达依据。 |

**Red Flags**
- 业务路径新增 `panic`、`fmt.Println`、`log.Println` 或敏感信息日志。
- `_ = err`、只打日志不 return、错误不带上下文。
- 新增 goroutine 但没有退出机制、取消上下文或 recover 策略。
- 新增第三方依赖但没有说明标准库或现有依赖为什么不足。

**Verification**
- 代码 diff 已按本规则检查，例外已记录。
- 受影响 package 已格式化并运行声明的构建或测试证据。
- `cc-review` 对命名、错误、日志、并发和依赖风险给出结论。

#### 约束等级
- 🚫 强制 — 必须遵守，违反会引入缺陷
- ⚠️ 警告 — 应避免，特殊情况需注明原因
- ✅ 验证 — 完成后自检

#### 1. 命名规范

**🚫 强制规则：**
| 对象 | 规范 | 禁止 |
|------|------|------|
| 文件名 | 全小写，下划线分隔 | `userService.go`, `UserService.go` |
| 结构体/接口 | 大驼峰，接口以 `er` 结尾 | 拼音、中英混拼 |
| 变量/函数 | 驼峰，导出用大写 | 拼音、中英混拼 |
| 常量 | 大驼峰或全大写下划线 | 拼音 |
| 测试文件 | `*_test.go`，函数以 `Test` 开头 | 其他命名 |

#### 2. 错误与异常处理

**🚫 强制规则：**
- 业务错误 → 必须使用自定义 Error 类型，通过返回值传递
- 禁止滥用 `panic`
- 错误透传 → 必须使用 `fmt.Errorf("... %w", err)` 包装
- 禁止吞掉错误（禁止 `_ = err` 或只打日志不 return）

**⚠️ 警告触发：**
- 独立 Goroutine 入口 → 必须使用 `defer recover()` 兜底
- 同一错误路径 → 应避免“记录日志后继续原样返回同一错误”，优先在边界层决定记录
- 返回错误时 → 应补充足够上下文，避免只返回底层原错

**✅ 验证清单：**
- [ ] 错误是否通过返回值传递而非 panic
- [ ] 错误是否使用 `%w` 包装保留堆栈
- [ ] 是否有 `_ = err` 或日志后无 return
- [ ] Goroutine 是否有 recover 兜底
- [ ] 是否存在重复日志 + 返回同一错误
- [ ] 返回错误是否包含足够上下文

#### 3. 日志规范

**🚫 强制规则：**
- 默认使用 `log/slog` 作为项目日志库
- 禁止新增 `fmt.Println`、`log.Println` 作为业务运行日志
- 应用启动时必须初始化统一 logger，并在模块间复用
- 日志格式必须至少包含：时间、等级、消息、源码文件名:行号、函数/方法名
- 时间精度默认要求到微秒
- 禁止在日志中打印用户敏感信息（手机号、身份证、银行卡等）

**⚠️ 警告触发：**
- 运行时日志默认建议落盘到日志文件；若改为 stdout/stderr + 平台采集，必须在 `.cc/context/project-context.md` 中说明
- 日志文件路径、切分策略、保留策略由使用者决定，但必须明确记录
- HTTP/gRPC 入口、异步任务、关键状态变更、外部调用失败应有可检索日志
- `slog.HandlerOptions` 若未开启源码位置信息（`AddSource`），需说明原因
- 若函数/方法名无法直接由日志库提供，需通过 wrapper 或 `With` 字段补齐

**✅ 验证清单：**
- [ ] 是否统一使用 `log/slog`
- [ ] 是否存在应用级 logger 初始化
- [ ] 日志是否包含时间、等级、消息、文件名:行号、函数/方法名
- [ ] 时间精度是否达到微秒
- [ ] Handler/Controller 入口是否打 INFO（含请求关键参数）
- [ ] 错误是否打 ERROR（含 traceID）
- [ ] 是否声明了日志落盘或等价采集方案
- [ ] 是否打印敏感信息

#### 4. 其他规范

**🚫 强制规则：**
- 魔法值必须定义为常量
- 写接口必须考虑幂等

**⚠️ 警告触发：**
- 并发场景 → 必须说明同步策略（`sync.Mutex` 或 `channel`）
- 控制流 → 优先使用早返回，避免无必要的 `else` 嵌套
- 设计抽象 → 默认使用具体类型，只有在测试替身、解耦或多实现确有需要时再引入接口
- 接口设计 → 优先自底向上，小接口优先（通常 1-2 个方法）
- 新增第三方依赖 → 必须说明标准库或现有依赖为什么不满足需求
- 新增 Goroutine → 必须说明退出机制；任务型并发优先使用 `errgroup.WithContext`

**✅ 验证清单：**
- [ ] 是否有魔法值未定义为常量
- [ ] 接口是否考虑了幂等
- [ ] 并发场景是否说明了同步策略
- [ ] 是否可通过早返回压平控制流
- [ ] 是否存在明显过早抽象
- [ ] 新接口是否足够小且有明确用途
- [ ] 新增第三方依赖是否有充分理由
- [ ] Goroutine 是否有明确退出机制
